Hi! I'm Quynh. I'm a junior web developer who has experience with HTML, CSS (SCSS and PostCSS), Javascript, Typescript, and NodeJS. I'm acquainted with design tools such as Adobe XD, Inkscape, and GIMP. I have work experience with the recent version of React, mostly front-end development. Currently, I'm learning C# in Unity. While working in the simulation industry and seeing how game development and web combined, I picked up an interest in learning Unity so that I can make something myself that I can call my own.

My hobbies include jogging, playing badminton, going wakeboarding, video games, boardgames, going to meetups, drawing, writing, watching anime, reading manga and webtoons, and playing with my dog Matcha.

Some Reflections:

  • Working with others’ codebase
  • HMIs present interesting problems for web development. It's very unconventional compared to developing typical websites. I picked up TypeScript on the way, while using inhouse software to connect it to the simulation database.
  • Keeping great documentation is a must. I learned the importance of READMEs.
  • Writing really ugly code, suffering for it later
  • It’s hard to see how things should be the first time on the job. People were unhappy about things, but I’m still grasping what should be and what shouldn’t be. Communication with the team, it’s easy to let each other go early before things get fully communicated (thoroughly)
  • Someone needs to be strict to you the first time. The person I was working with had a very peculiar way of doing things, however he was open minded enough to consider that I was a junior. It’s good having time to explore, but could also be easy to get lost when you’re trying out too many things at a time, not to mention, make the codebase very messy and inconsistent. So I followed what was basically his style. We have a programming styling guide, but it seems to have been loosely referred to. The team was small and shrinking. Sometimes it was hard getting a hold of others. I have to adjust my schedule and set an appointment ahead of time. Too much information means no information. Outdated information means no trust of information. Documentation is where information goes to die… this is not very ideal. I get that people don’t want to step on each other’s foot, but also it’s just split sources we can refer to in many places which can be very disheartening.
  • Understanding what you don’t understand. Being able to pinpoint those things. It’s still something I’m struggling to improve on.
  • I’m usually pretty slow, but I learn not to take things personally. I think I need to work on dealing with someone crossing the lines no matter how above seniority they are. Learning new technology.
  • Being interested in how the code works under the hood. That’s hard to do in the beginning when you're just trying to get things onto the screen to behave the way they should, but once you pick up patterns from common problems, there’ll be a growing fascination with how it works on a deeper level.
